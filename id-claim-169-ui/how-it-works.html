How It Works

A deep dive into the six-step pipeline from raw identity data to a verifiable, tamper-proof QR code — and how it is verified in the field.

The Encoding Pipeline

01

Prepare Identity Data

Convert JSON identity information — demographics, biometrics, and metadata — into Claim 169 CBOR structure.

JSON → CBOR Map
02

Generate CWT

Create a CBOR Web Token (CWT) using the Claim 169 formatted identity data with all required fields.

CWT (RFC 8392)
03

Sign with COSE

Sign the CWT using COSE Sign1 with ED25519 or ECC algorithm per RFC 8152. Signing is mandatory.

COSE Sign1 / RFC 8152
04

Compress

Apply zlib or Brotli compression to the signed token. Verifiers detect format via the zlib magic number.

zlib / Brotli
05

Encode Base45

Convert the compressed CWT using Base45 encoding, optimized for QR code character sets.

Base45 Encoding
06

Generate QR Code

Render the Base45-encoded string as a machine-readable QR code for printing on physical identity cards.

Security Architecture

Cryptographic assurance without real-time server dependency

Mandatory Signing

— Claim 169 Specification

  • Biometric Encryption

    Sensitive biometric data should be encrypted using COSE_Encrypt/COSE_Encrypt0 mechanisms. If encryption is used, sign first then encrypt the result.

  • Key Discovery

    In v1.2.1, JWKS-based .well-known endpoints are replaced with COSE-based key discovery mechanisms. Public keys must be validated against preconfigured keys.

  • Nested CWT Operations

    Producers normally should sign the message and then encrypt the result when encryption is necessary, following standard CWT nested token procedures.
